👉 당신이 알아야 하는 건 3개다

1. 시스템이 멈췄는지
2. 상태가 꼬였는지
3. 실행이 비정상인지

## 2️⃣ 감지 방법 (실전 기준)

### 🔹 1. 로그 끊김 (가장 빠른 신호)

확인 파일:  EVIDENCE/logs/*.log

정상: 매 분 / 매 실행마다 로그 찍힘

비정상: 로그가 멈춤 = 시스템 멈춤


👉 기준: * 2~3분 이상 로그 없음 → 이상

### 🔹 2. state.json 시간 체크

확인: STATE/state.json

핵심 필드: last_update_time


비정상: 현재 시간 - last_update_time > 실행 주기

예: * 1분 전략인데 5분 업데이트 없음 → 문제

### 🔹 3. 계좌 vs state 불일치

확인: * 실제 계좌 (LS / Upbit)  * state.json

비정상: 계좌: 보유 있음
state: 없음

👉 position_reconciler 문제 또는 중단 흔적

🔹 4. 주문 흔적 체크

확인: EVIDENCE/ledger/

비정상: 주문 기록 없음 또는 중복 주문 발생
`
### 🔹 5. scheduler 상태 확인

확인: un.py 로그

비정상: 계속 NO_ENTRY 또는 FORCE_EXIT 반복

👉 시간/모드 오류

## 3️⃣ 가장 실전적인 방법 (핵심)

👉 “3줄 체크”로 끝낸다

1. 로그 살아있는가?
2. state 시간 최신인가?
3. 계좌와 state 일치하는가?

👉 이 3개면 90% 문제 잡는다

## 4️⃣ 당신 구조 기준 위험 신호

이거 나오면 바로 중단 판단:

- 로그 멈춤
- state 갱신 안됨
- 동일 종목 중복 진입
- 포지션 수 초과
- 거래 안되는데 계속 시도

## 5️⃣ 절대 하면 안 되는 대응

❌ 코드 수정하면서 돌림
❌ 그냥 재실행
❌ 원인 없이 계속 실행

👉 이게 계좌 터지는 패턴

## 6️⃣ 올바른 대응 흐름

1. 실행 중지
2. state.json 확인
3. 실제 계좌 확인
4. reconcile 필요 여부 판단
5. 원인 확인 후 재시작

## 7️⃣ 한 줄 핵심

👉 **흔적(log / state / 계좌)**로 직접 판단한다”
