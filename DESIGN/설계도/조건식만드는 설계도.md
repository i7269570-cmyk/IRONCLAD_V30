1. IRONCLAD 실행 규격과 1:1 호환

가장 중요합니다.

config/strategies/*.yaml
→ 나중에 IRONCLAD의 strategy_spec.yaml / selector_rules.yaml / entry_rules.yaml / exit_rules.yaml과
구조가 호환되게 설계

즉, 연구소에서 찾은 전략은 이렇게 흘러야 합니다.

TRINITY_RESEARCH 결과
→ 수정 최소화
→ IRONCLAD STRATEGY로 이식
→ 바로 실행

연구 구조와 실행 구조가 다르면, 마지막에 또 다 뜯어고쳐야 해서 망합니다.

2. metrics.py를 “실전 기준”으로 강화

단순 수익률만 보면 안 됩니다.
반드시 아래를 같이 봐야 합니다.

- 슬리피지 반영
- 수수료 반영
- max_daily_loss 초과 시 강제 종료 시뮬레이션
- FORCE_EXIT 시간 반영
- max_positions / max_exposure 반영

즉, 연구 단계부터:

“예쁘게 보이는 전략”이 아니라
“실전에서 살아남는 전략”만 통과

시키는 구조입니다.

3. 실시간 데이터 파이프라인 연결

이미 3대 PC로 데이터를 수집하고 있으니, 연구소와 이어줘야 합니다.

scripts/fetch_live_data.py

역할은 이것입니다.

수집된 원시 데이터
→ data/raw/stock 또는 data/raw/crypto로 자동 적재
→ cleaners / transformers 거쳐 processed로 이동

이게 있어야 연구 흐름이 끊기지 않습니다.

4. indicators를 금융 도메인으로 분리

지금은 금융 전략 연구소이지만, 나중에 다른 분야로 확장할 수 있으니:

indicators/financial/

로 분리한 게 맞습니다.

이렇게 해야 나중에:

go/
real_estate/

같은 다른 자동화 공장으로 확장해도 충돌이 없습니다.

5. experiments 재현성 확보

각 실험 폴더에 반드시 남겨야 하는 것:

- config_snapshot.yaml
- seed.txt
- run_info.json

이게 없으면 나중에 “왜 이 결과가 나왔지?”가 안 풀립니다.

6. output은 타임스탬프 기반 관리

이건 필수입니다.

YYYYMMDD_HHMM_전략명

형식으로 저장해야, 결과가 쌓여도 관리가 됩니다.

역할을 다시 한 줄씩 정리하면
src = 연구 코드
data = 연구 데이터
config = 실험 설정
experiments = 실험 단위 기록
output = 결과 저장
scripts = 실행 스크립트
tests = 검증
docs = 의사결정 기록
가장 중요한 분리 원칙

TRINITY_RESEARCH에서 하는 일
- 조건식 찾기
- 수치 탐색
- 백테스트
- 실험 비교
- 전략 채택/폐기
